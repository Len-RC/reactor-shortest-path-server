# 面试问答 - 多线程和网络问题

## 多线程相关问题

### Q1: 你的线程池是怎么实现的？任务队列用的什么数据结构？为什么？

**答案**：

我的线程池实现包含三个核心组件：
1. **工作线程数组**（std::vector<std::thread>）：固定 4 个工作线程
2. **任务队列**（std::queue<std::function<void()>>）：存储待执行的任务
3. **同步机制**：mutex + condition_variable

**为什么用 std::queue？**
- FIFO 先进先出，保证任务按提交顺序执行，公平性好
- 操作简单：push 入队，pop 出队，front 取队首
- 性能足够：单端入队出队，O(1) 时间复杂度

**工作流程**：
1. 主线程调用 submit() 提交任务到队列，notify_one() 唤醒一个工作线程
2. 工作线程在 cv.wait() 等待，被唤醒后从队列取任务执行
3. 执行完继续等待下一个任务

**代码关键点**：
```cpp
// 提交任务
void submit(T&& t) {
    {
        std::unique_lock<std::mutex> lock(mtx);
        tasks.emplace(std::forward<T>(t));  // 完美转发，避免拷贝
    }
    cv.notify_one();  // 唤醒一个工作线程
}

// 工作线程循环
cv.wait(lock, [this]() { return stop || !tasks.empty(); });
```

### Q2: 线程池的工作线程如果都在忙，新任务来了怎么办？会阻塞还是拒绝？

**答案**：


**当前实现**：任务会被放入队列，不会阻塞也不会拒绝。

**详细说明**：
- submit() 只是把任务放入队列就返回，不等待执行
- 队列是无界的（std::queue 默认基于 deque），理论上可以无限增长
- 工作线程忙时，任务在队列中排队等待

**优点**：
- 提交任务的线程（I/O 线程）不会阻塞，可以继续处理其他连接
- 实现简单

**缺点**：
- 高并发下队列可能无限增长，内存爆炸
- 没有背压（backpressure）机制

**改进方案**：
1. **有界队列 + 拒绝策略**：队列满了拒绝新任务或阻塞
2. **动态线程池**：忙时增加线程，闲时减少
3. **优先级队列**：重要任务优先执行

**实际回答**：
"当前实现中，任务会放入无界队列排队，不会阻塞或拒绝。但生产环境中应该加入有界队列和拒绝策略，防止内存爆炸。比如队列达到 1000 个任务时，拒绝新任务并返回错误给客户端。"

### Q3: 你提到用了 mutex 和 condition_variable，能说说它们的作用吗？为什么不用 spinlock？

**答案**：

**mutex（互斥锁）的作用**：
- 保护任务队列，防止多个线程同时访问导致数据竞争
- 提交任务时加锁，取任务时加锁

**condition_variable（条件变量）的作用**：
- 工作线程在队列为空时等待，避免忙等（busy waiting）
- 有新任务时唤醒工作线程，避免轮询

**为什么不用 spinlock（自旋锁）？**


| 特性 | mutex + cv | spinlock |
|------|-----------|----------|
| 等待方式 | 线程休眠，让出 CPU | 忙等，持续占用 CPU |
| 适用场景 | 等待时间长（如队列为空） | 等待时间极短（几个 CPU 周期） |
| CPU 占用 | 低 | 高 |
| 上下文切换 | 有（休眠/唤醒） | 无 |

**我的场景分析**：
- 工作线程可能长时间等待（队列为空时）
- 任务执行时间不确定（查询数据库可能几毫秒到几十毫秒）
- 使用 mutex + cv 可以让线程休眠，节省 CPU

**spinlock 适用场景**：
- 临界区极短（几条指令）
- 多核系统，锁竞争不激烈
- 实时系统，不能容忍上下文切换延迟

### Q4: 工作线程通过管道通知 Reactor，为什么不直接用 condition_variable 唤醒？

**答案**：

**为什么用管道而不是 condition_variable？**

**原因一：Reactor 在 epoll_wait 阻塞**
- Reactor 线程在 epoll_wait() 阻塞，等待 I/O 事件
- condition_variable 无法唤醒在 epoll_wait 中的线程
- 管道的读端注册到 epoll，写入管道会触发 EPOLLIN 事件，唤醒 epoll_wait

**原因二：统一事件模型**
- Reactor 模式的核心是事件驱动
- 所有事件（网络 I/O、管道通知）都通过 epoll 统一处理
- 保持架构一致性

**工作流程**：
```
工作线程处理完任务
  ↓
把结果放入消息队列
  ↓
往管道写端写 1 字节（非阻塞）
  ↓
epoll_wait 返回（管道读端可读）
  ↓
Reactor 从消息队列取结果
  ↓
调用 write 发送给客户端
```

**如果用 condition_variable 会怎样？**
- 需要两套机制：epoll 处理 I/O，cv 处理任务完成通知
- Reactor 需要在 epoll_wait 和 cv.wait 之间切换，复杂且低效
- 无法统一处理所有事件

### Q5: 你的代码里哪些地方有竞态条件（race condition）？怎么保证线程安全的？

**答案**：

**竞态条件出现的地方**：

**1. 任务队列（tasks）**
- **竞态**：多个 I/O 线程同时 submit()，多个工作线程同时取任务
- **保护**：用 mutex 加锁，submit 和取任务都在锁保护下进行

**2. 消息队列（pending_msgs_）**
- **竞态**：多个工作线程同时往队列放结果
- **保护**：用 mutex 加锁

**3. 双缓冲的版本号（_currentVersion）**
- **竞态**：查询线程读版本号，更新线程写版本号
- **保护**：用 atomic<int>，原子操作，无需加锁

**4. Redis 连接池和 MySQL 连接池**
- **竞态**：多个工作线程同时获取/归还连接
- **保护**：用 mutex + condition_variable

**线程安全保证的关键点**：

```cpp
// 1. 任务队列加锁
void submit(T&& t) {
    {
        std::unique_lock<std::mutex> lock(mtx);  // 加锁
        tasks.emplace(std::forward<T>(t));
    }  // 自动解锁
    cv.notify_one();
}

// 2. 双缓冲用 atomic
std::atomic<int> _currentVersion{0};
int version = _currentVersion.load(std::memory_order_acquire);  // 原子读
_currentVersion.store(backup, std::memory_order_release);       // 原子写

// 3. 连接池加锁
std::shared_ptr<MYSQL> getConnection() {
    std::unique_lock<std::mutex> lock(_queueMutex);
    cv_out.wait(lock, [this](){return !_connectionQue.empty();});
    // ...
}
```

**没有竞态的地方**：
- 每个连接的 in_buffer_（每个连接独立，不共享）
- epoll 的 fd（内核保证线程安全）

### Q6: 双缓冲用了 atomic 变量，为什么不用 mutex？atomic 的 memory_order 你了解吗？

**答案**：

**为什么用 atomic 而不是 mutex？**


**性能对比**：

| 操作 | atomic | mutex |
|------|--------|-------|
| 读版本号 | 无锁，几个 CPU 周期 | 加锁/解锁，几十到几百个周期 |
| 写版本号 | 无锁，原子操作 | 加锁/解锁 |
| 并发读 | 完全并发，无竞争 | 串行化，有竞争 |

**我的场景**：
- 查询非常频繁（12w QPS），每次查询都要读版本号
- 更新很少（可能几分钟或几小时一次）
- 读多写少，atomic 性能远超 mutex

**memory_order 说明**：

```cpp
// 读取版本号
int version = _currentVersion.load(std::memory_order_acquire);
```
- **memory_order_acquire**：保证后续的读操作不会被重排到这条指令之前
- 读到版本号后，访问对应版本的缓存，保证看到的是完整的数据

```cpp
// 切换版本号
_currentVersion.store(backup, std::memory_order_release);
```
- **memory_order_release**：保证之前的写操作（加载缓存数据）不会被重排到这条指令之后
- 确保缓存数据完全加载后，才切换版本号

**acquire-release 语义**：
- release 写入的数据，acquire 读取时能看到
- 形成同步点，保证内存可见性
- 比 seq_cst（顺序一致性）性能更好

**简化回答**：
"用 atomic 是因为读多写少，查询时无锁读取版本号，性能远超 mutex。memory_order_acquire 保证读到版本号后能看到完整的缓存数据，memory_order_release 保证缓存加载完才切换版本号，形成同步点。"

## 网络相关问题

### Q7: Epoll 的边缘触发（ET）和水平触发（LT）有什么区别？为什么选择 ET？

**答案**：

**水平触发（Level Triggered, LT）**：
- 只要 fd 可读/可写，epoll_wait 就会返回
- 类似"状态触发"：有数据就通知

**边缘触发（Edge Triggered, ET）**：
- 只在 fd 状态变化时通知一次
- 类似"事件触发"：从不可读变为可读时通知


**举例说明**：

假设 socket 缓冲区有 1000 字节数据：

| 操作 | LT 行为 | ET 行为 |
|------|---------|---------|
| 第一次 epoll_wait | 返回可读 | 返回可读 |
| read 500 字节 | - | - |
| 第二次 epoll_wait | 返回可读（还有 500 字节） | 不返回（状态没变化） |
| 新数据到达 | 返回可读 | 返回可读（状态变化） |

**为什么选择 ET？**

**优点**：
1. **减少系统调用**：LT 模式下，如果没读完，每次 epoll_wait 都返回，浪费 CPU
2. **更高效**：ET 强制一次读完，减少 epoll_wait 调用次数
3. **更适合高并发**：大量连接时，LT 会频繁返回，ET 只在有新事件时返回

**代价**：
- 必须用非阻塞 I/O
- 必须循环读取直到 EAGAIN，否则会丢数据

**我的实现**：
```cpp
// 注册边缘触发
ev.events = EPOLLIN | EPOLLET;

// 循环读取直到 EAGAIN
void handleRead() override {
    while (true) {
        ssize_t n = recv(getFd(), buf, sizeof(buf), 0);
        if (n > 0) {
            in_buffer_.append(buf, n);
        } else if (n == 0) {
            // 连接关闭
            handleClose();
            return;
        } else {
            if (errno == EAGAIN || errno == EWOULDBLOCK) {
                break;  // 读完了
            }
            // 其他错误
            handleClose();
            return;
        }
    }
}
```

### Q8: 非阻塞 I/O 下，read 返回 EAGAIN 是什么意思？你是怎么处理的？

**答案**：

**EAGAIN 的含义**：
- "Resource temporarily unavailable"（资源暂时不可用）
- 在非阻塞模式下，表示当前没有数据可读，但不是错误
- 如果是阻塞模式，read 会一直等待；非阻塞模式立即返回 EAGAIN

**我的处理方式**：

```cpp
ssize_t n = recv(getFd(), buf, sizeof(buf), 0);
if (n > 0) {
    // 读到数据，继续处理
    in_buffer_.append(buf, n);
} else if (n == 0) {
    // 对端关闭连接
    handleClose();
    return;
} else {  // n < 0
    if (errno == EAGAIN || errno == EWOULDBLOCK) {
        // 数据读完了，正常情况，退出循环
        break;
    } else {
        // 真正的错误（如连接重置）
        handleClose();
        return;
    }
}
```

**为什么要循环读取直到 EAGAIN？**
- ET 模式下，只在状态变化时通知一次
- 如果不读完，下次 epoll_wait 不会再通知
- 必须循环读取直到 EAGAIN，确保所有数据都读完

**EWOULDBLOCK vs EAGAIN**：
- 在 Linux 上，两者值相同
- 都表示"暂时没数据"
- 为了兼容性，两个都检查

### Q9: TCP 的粘包问题你遇到过吗？HTTP 和 TCP 协议你是怎么区分和解析的？

**答案**：

**TCP 粘包问题**：

TCP 是字节流协议，没有消息边界。可能出现：
1. **粘包**：多个消息粘在一起（"msg1msg2"）
2. **拆包**：一个消息被拆成多次接收（"msg" + "1"）

**我的解决方案**：

**TCP 协议**：用换行符 `\n` 作为消息边界
```cpp
// 循环查找换行符
while (true) {
    size_t pos = in_buffer_.find('\n');
    if (pos == std::string::npos) 
        break;  // 没有完整消息，等待更多数据
    
    std::string line = in_buffer_.substr(0, pos + 1);
    in_buffer_.erase(0, pos + 1);  // 从缓冲区移除
    // 处理 line
}
```

**HTTP 协议**：用 `\r\n\r\n` 标识请求头结束
```cpp
size_t header_end = in_buffer_.find("\r\n\r\n");
if (header_end == std::string::npos) {
    return false;  // 还没收到完整HTTP头
}
```

**协议区分**：

```cpp
// 1. 检查前几个字节
if (in_buffer_.size() >= 4 && 
    (in_buffer_.substr(0, 4) == "GET " || 
     in_buffer_.substr(0, 5) == "POST ")) {
    is_http_ = true;
} else if (in_buffer_[0] != 'G' && in_buffer_[0] != 'P') {
    is_http_ = false;  // 肯定不是 HTTP
}

// 2. 根据协议类型解析
if (is_http_) {
    parseHttpRequest();
} else {
    // 按行解析 TCP 消息
}
```

**关键点**：
- 用缓冲区（in_buffer_）累积数据
- 找到完整消息才处理，否则继续等待
- 协议判断只做一次（http_parsed_ 标志）

### Q10: Keep-Alive 是什么？为什么能提升性能？你的代码里是怎么实现的？

**答案**：

**Keep-Alive 是什么？**

HTTP/1.0 默认每个请求都要建立新连接：
```
客户端 → 三次握手 → 服务器
客户端 → 发送请求 → 服务器
客户端 ← 返回响应 ← 服务器
客户端 → 四次挥手 → 服务器
```

HTTP/1.1 的 Keep-Alive 允许复用连接：
```
客户端 → 三次握手 → 服务器
客户端 → 请求1 → 服务器
客户端 ← 响应1 ← 服务器
客户端 → 请求2 → 服务器  // 复用连接
客户端 ← 响应2 ← 服务器
...
客户端 → 四次挥手 → 服务器
```

**性能提升原因**：


1. **减少握手开销**：三次握手 + 四次挥手需要 7 个数据包，延迟高
2. **减少 TIME_WAIT 状态**：每次关闭连接会进入 TIME_WAIT（2MSL，通常 60 秒），占用端口
3. **减少 CPU 开销**：创建/销毁连接需要内核资源分配
4. **提高吞吐量**：连接建立后立即可用，无需等待握手

**我的实现**：

```cpp
// HTTP 响应头中设置 Keep-Alive
std::ostringstream response;
response << "HTTP/1.1 200 OK\r\n"
         << "Content-Type: text/plain; charset=utf-8\r\n"
         << "Content-Length: " << result.length() << "\r\n"
         << "Connection: keep-alive\r\n"           // 保持连接
         << "Keep-Alive: timeout=60, max=100\r\n"  // 60秒超时，最多100个请求
         << "\r\n"
         << result;
```

**服务端处理**：
- 发送响应后不关闭连接
- 继续在 epoll 中监控这个 fd
- 客户端可以继续发送请求

**Nginx 配置**：
```nginx
upstream shortest_path_cluster {
    keepalive 512;  // Nginx 到后端保持 512 个长连接
}
```

**性能数据**：
- 无 Keep-Alive：QPS ~3w（频繁握手）
- 有 Keep-Alive：QPS 12w+（连接复用）

### Q11: 如果客户端突然断开连接（比如网络故障），服务端怎么知道？会有什么问题？

**答案**：

**服务端如何检测断开？**

**方式一：read 返回 0**
```cpp
ssize_t n = recv(getFd(), buf, sizeof(buf), 0);
if (n == 0) {
    // 对端正常关闭（发送了 FIN）
    handleClose();
    return;
}
```

**方式二：read 返回错误**
```cpp
if (n < 0) {
    if (errno == ECONNRESET) {
        // 连接被重置（对端异常断开）
        handleClose();
    }
}
```

**方式三：write 失败**
```cpp
ssize_t n = send(getFd(), data, len, 0);
if (n < 0) {
    if (errno == EPIPE || errno == ECONNRESET) {
        // 对端已关闭，写入失败
        handleClose();
    }
}
```

**可能的问题**：


**1. 半开连接（Half-Open Connection）**
- 客户端断网，服务端不知道
- 服务端的 fd 一直占用，浪费资源
- 解决：设置 TCP keepalive 或应用层心跳

**2. 工作线程正在处理任务**
- 客户端断开，但工作线程还在查询数据库
- 查询完发送响应时才发现连接已关闭
- 解决：任务执行前检查连接是否有效（可选）

**3. 资源泄漏**
- 如果 handleClose() 没正确清理，会泄漏 fd、内存
- 解决：RAII，确保资源自动释放

**我的处理**：
```cpp
void handleClose() override {
    std::cout << "客户端断开连接，fd: " << getFd() << "\n";
    getLoop()->removeHandler(this);  // 从 epoll 移除
    delete this;  // 释放资源（析构函数会 close(fd)）
}
```

**改进方案**：
- 设置 TCP keepalive：`setsockopt(fd, SOL_SOCKET, SO_KEEPALIVE, ...)`
- 应用层心跳：定期发送 ping，超时未响应则关闭
- 超时机制：连接空闲超过 N 秒自动关闭

### Q12: 你提到管道缓冲区是 64KB，这个值是怎么来的？能调整吗？

**答案**：

**64KB 是 Linux 默认管道缓冲区大小**

可以通过以下方式查看：
```bash
# 查看系统默认值
cat /proc/sys/fs/pipe-max-size
# 通常是 1048576 字节（1MB），但实际分配是 64KB

# 查看当前管道大小
fcntl(pipe_fd[0], F_GETPIPE_SZ)
```

**为什么是 64KB？**
- Linux 内核默认值，平衡内存占用和性能
- 对于大多数应用足够（每次写入通常几百字节）
- 太大浪费内存，太小容易满

**能调整吗？**

可以，使用 `fcntl` 调整：
```cpp
// 设置管道大小为 1MB
fcntl(pipe_fd[1], F_SETPIPE_SZ, 1024 * 1024);
```

**我的场景分析**：
- 每次写入 1 字节（只是通知信号）
- 高并发下可能瞬间写入大量通知
- 64KB 可以容纳 65536 个通知，通常足够
- 如果还不够，说明 Reactor 处理太慢，应该优化 Reactor 而不是扩大管道

**为什么会满？**
- 工作线程写入速度 > Reactor 读取速度
- Reactor 被阻塞（比如发送响应时阻塞）
- 解决：管道写端设置非阻塞，写入失败时重试

## 综合问题

### Q13: 如果某个客户端恶意发送大量请求，会不会把你的服务打挂？有什么防护措施？

**答案**：

**当前实现的脆弱性**：


**1. 任务队列无界**
- 恶意客户端疯狂发请求，任务队列无限增长
- 内存爆炸，OOM（Out of Memory）

**2. 连接数无限制**
- 可以建立大量连接，耗尽 fd（文件描述符）
- 系统默认限制 1024 或 65535

**3. 无请求频率限制**
- 单个连接可以无限发送请求
- 占用所有工作线程

**防护措施**：

**应用层防护**：

1. **限制任务队列大小**
```cpp
void submit(T&& t) {
    std::unique_lock<std::mutex> lock(mtx);
    if (tasks.size() >= MAX_QUEUE_SIZE) {
        throw std::runtime_error("任务队列已满");
    }
    tasks.emplace(std::forward<T>(t));
}
```

2. **限制连接数**
```cpp
static std::atomic<int> connection_count{0};

void handleRead() override {
    if (connection_count.load() >= MAX_CONNECTIONS) {
        // 拒绝新连接
        close(connfd);
        return;
    }
    connection_count++;
}
```

3. **请求频率限制（令牌桶算法）**
```cpp
// 每个连接维护一个令牌桶
struct TokenBucket {
    int tokens;
    time_t last_refill;
    
    bool consume() {
        refill();  // 定期补充令牌
        if (tokens > 0) {
            tokens--;
            return true;
        }
        return false;  // 令牌耗尽，拒绝请求
    }
};
```

4. **超时机制**
```cpp
// 连接空闲超过 60 秒自动关闭
time_t last_active;
if (time(NULL) - last_active > 60) {
    handleClose();
}
```

**系统层防护**：

1. **ulimit 限制**
```bash
# 限制进程最大 fd 数
ulimit -n 10000
```

2. **iptables 防火墙**
```bash
# 限制单 IP 连接数
iptables -A INPUT -p tcp --dport 9090 -m connlimit --connlimit-above 100 -j REJECT
```

**Nginx 层防护**：

```nginx
# 限制单 IP 请求频率
limit_req_zone $binary_remote_addr zone=one:10m rate=100r/s;
limit_req zone=one burst=200 nodelay;

# 限制连接数
limit_conn_zone $binary_remote_addr zone=addr:10m;
limit_conn addr 10;
```

**实际回答**：
"当前实现确实有被打挂的风险。生产环境中，我会在应用层加入任务队列大小限制、连接数限制、请求频率限制。同时依赖 Nginx 做第一层防护，限制单 IP 的请求频率和连接数。这样多层防护，可以有效防止恶意攻击。"

### Q14: 你的连接池初始 4 个，最大 8 个，这个数字是怎么定的？如何调优？

**答案**：

**当前配置**：
- MySQL 连接池：初始 4，最大 8
- Redis 连接池：初始 4，最大 8
- 线程池：固定 4 个工作线程

**这些数字的来源**：

**1. 基于 CPU 核心数**
- 我的测试服务器是 4 核
- 线程池 4 个线程，充分利用 CPU，避免过多上下文切换
- 连接池初始 4 个，与线程数匹配

**2. 经验值**
- 初始连接数 = 线程数（避免线程等待连接）
- 最大连接数 = 2 × 线程数（应对突发流量）

**如何调优？**

**调优原则**：

1. **监控指标**
```
- 连接池等待时间：如果经常等待，说明连接不够
- 连接池利用率：如果利用率低，说明连接太多
- 数据库负载：如果数据库 CPU/内存高，减少连接数
```

2. **压测调优**
```bash
# 逐步增加并发，观察性能
ab -n 100000 -c 100 ...   # 100 并发
ab -n 100000 -c 500 ...   # 500 并发
ab -n 100000 -c 1000 ...  # 1000 并发
```

3. **根据场景调整**

| 场景 | 初始连接 | 最大连接 | 理由 |
|------|---------|---------|------|
| 低并发 | 2-4 | 4-8 | 节省资源 |
| 高并发 | 8-16 | 32-64 | 减少等待 |
| 数据库性能差 | 4-8 | 8-16 | 避免压垮数据库 |
| 缓存命中率高 | 2-4 | 4-8 | 很少访问数据库 |

**我的调优过程**：

```
初始配置：4 个连接
  ↓
压测 1000 并发，发现连接池等待时间长
  ↓
增加到最大 8 个连接
  ↓
压测 2000 并发，性能提升，等待时间降低
  ↓
继续增加到 16 个，性能没提升（瓶颈在网络层）
  ↓
最终选择：初始 4，最大 8
```

**动态调整**：
```cpp
// 根据负载动态调整（生产环境可选）
if (wait_time > 100ms && current_size < max_size) {
    // 创建新连接
} else if (idle_time > 60s && current_size > init_size) {
    // 关闭空闲连接
}
```

**实际回答**：
"初始 4 个是基于 4 核 CPU，与线程数匹配。最大 8 个是经验值，2 倍线程数。实际调优时，我通过压测观察连接池等待时间和利用率，发现 8 个连接足够应对 2000 并发，再增加性能也不提升，说明瓶颈在网络层。"

### Q15: 如果 Redis 挂了，系统还能工作吗？有降级方案吗？

**答案**：

**当前实现的问题**：

Redis 挂了，系统会**完全不可用**：
```cpp
auto redis_conn = RedisPool::getInstance().getConnection();
if (!redis_conn) {
    std::cerr << "获取Redis连接失败，降级到MySQL\n";
    return "服务暂时不可用\n";  // 直接返回错误
}
```

**降级方案**：

**方案一：降级到 MySQL（推荐）**


```cpp
std::string handleClientRequest(const std::string& client_input) {
    // 1. 先尝试 Redis
    auto redis_conn = RedisPool::getInstance().getConnection();
    if (redis_conn) {
        // 从 Redis 查询
        return queryFromRedis(redis_conn, start_id, end_id);
    }
    
    // 2. Redis 不可用，降级到 MySQL
    std::cerr << "Redis 不可用，降级到 MySQL\n";
    auto mysql_conn = getConnection();
    if (!mysql_conn) {
        return "服务暂时不可用\n";
    }
    
    // 从 MySQL 查询
    return queryFromMySQL(mysql_conn, start_id, end_id);
}
```

**优点**：
- 系统仍然可用，只是性能下降
- 延迟从 4-9ms 增加到 20-50ms
- QPS 从 12w 降到 2-5w

**缺点**：
- MySQL 压力突然增大，可能扛不住
- 需要提前做好 MySQL 容量规划

**方案二：使用内存缓存兜底**

```cpp
// 启动时预加载热点数据到内存
std::unordered_map<long long, PathResult> hot_cache;

std::string handleClientRequest(const std::string& client_input) {
    // 1. 先查内存缓存
    auto it = hot_cache.find(makeKey(start_id, end_id));
    if (it != hot_cache.end()) {
        return it->second;
    }
    
    // 2. 再查 Redis
    auto redis_conn = RedisPool::getInstance().getConnection();
    if (redis_conn) {
        return queryFromRedis(redis_conn, start_id, end_id);
    }
    
    // 3. 最后降级到 MySQL
    return queryFromMySQL(mysql_conn, start_id, end_id);
}
```

**优点**：
- 热点数据仍然高性能
- 减轻 MySQL 压力

**缺点**：
- 内存占用大
- 缓存更新复杂

**方案三：熔断降级**

```cpp
class CircuitBreaker {
    int failure_count = 0;
    bool is_open = false;
    
    bool allowRequest() {
        if (is_open) {
            // 熔断打开，拒绝请求
            return false;
        }
        return true;
    }
    
    void recordFailure() {
        failure_count++;
        if (failure_count > 10) {
            is_open = true;  // 打开熔断器
            // 60 秒后尝试恢复
        }
    }
};

// 使用熔断器
if (!circuit_breaker.allowRequest()) {
    // 直接降级到 MySQL，不再尝试 Redis
    return queryFromMySQL(mysql_conn, start_id, end_id);
}
```

**Redis 高可用方案**：

1. **Redis 主从复制**
```
Master (写)
  ↓ 复制
Slave1 (读)
Slave2 (读)
```

2. **Redis Sentinel（哨兵）**
- 监控 Master 健康状态
- Master 挂了自动切换到 Slave
- 客户端自动发现新 Master

3. **Redis Cluster（集群）**
- 数据分片，多个 Master
- 每个 Master 有 Slave 备份
- 高可用 + 高性能

**实际回答**：
"当前实现中，Redis 挂了系统会不可用。生产环境中，我会实现降级方案：Redis 不可用时降级到 MySQL，虽然性能下降但系统仍然可用。同时部署 Redis 主从复制 + Sentinel，实现 Redis 高可用，Master 挂了自动切换到 Slave。"
